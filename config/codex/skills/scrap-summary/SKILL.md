# スクラップ要約

## 目的

毎日スクラップしている記事を確認しやすくするため、対象日分の `~/MyLife/Clippings/` 配下のファイルを要約し、対応するデイリーノート `~/MyLife/pages/YYYY_MM_DD.md` に `## 要約` セクションとして追記または置換する。

## 基本方針

- 日付判定は Asia/Tokyo 基準にする。
- ユーザーが日付を指定しない場合は、実行日の前日を対象日にする。
- 対象は `~/MyLife/Clippings/` 直下にあり、ファイル名の先頭が対象日の `YYYYMMDD` で始まるファイルだけにする。
- 対象外の日付や別ディレクトリのファイルは処理しない。
- 自動化や定期実行は、ユーザーが明示的に依頼した場合だけ作成する。
- 読み取れないファイルも、可能な範囲でタイトル・URL・要約欄に「不明」や読み取り状況を記録する。

## ワークフロー

1. 対象日を決める。
   - 日付指定がなければ、Asia/Tokyoで「今日の前日」。
   - 例: 実行日が `2026-04-28` なら対象日は `2026-04-27`、ファイル接頭辞は `20260427`、デイリーノートは `~/MyLife/pages/2026_04_27.md`。

2. 対象ファイルを列挙する。
   - `find ~/MyLife/Clippings -maxdepth 1 -type f -name 'YYYYMMDD*' -print | sort`
   - 件数を確認し、0件ならデイリーノートに空の要約を作らず、対象ファイルなしを報告する。

3. 各ファイルから情報を抽出する。
   - Markdown frontmatter がある場合は、`title:` をタイトル、`source:` をURLとして優先する。
   - frontmatter が欠ける場合は、本文中の最初の見出し、リンク、URLを手がかりにする。
   - HTMLやMarkdownは本文テキストを抽出して読む。画像、表、iframe、ナビゲーション、SNS埋め込みなどは要約対象の主情報でない限り無視する。
   - 抽出できないタイトル・URL・本文は「不明」とする。

4. 記事ごとに2〜4文で要約する。
   - 主題、重要点、読む価値があるポイントを含める。
   - 断定しすぎず、記事本文から読み取れる範囲で中立的にまとめる。
   - 同一URLや同名ファイルが複数ある場合も、対象日のファイルとして記録する。明らかに重複なら要約欄に「重複ファイル」と明記してよい。

5. デイリーノートの `## 要約` セクションを更新する。
   - `~/MyLife/pages/YYYY_MM_DD.md` が存在しない場合は作成してよい。
   - 既に `## 要約` セクションがある場合は、重複追記せず、そのセクション全体を置き換える。
   - 既存セクションの範囲は、`## 要約` から次の同レベル見出し `## ` の直前まで。次の `## ` がなければファイル末尾まで。
   - `## 要約` がなければファイル末尾に追加する。

## 出力形式

通常テキストのみで、画像や表は使わない。

```markdown
## 要約

### タイトル
- URL: ...
- 要約: ...
```

記事ごとにこのブロックを繰り返す。タイトルが不明な場合は `### 不明` とする。

## 確認

更新後に次を確認する。

- 対象日の `YYYYMMDD*` ファイルだけを処理したこと。
- `~/MyLife/pages/YYYY_MM_DD.md` に `## 要約` が1つだけ存在すること。
- 記事件数が対象ファイル数と対応していること。重複ファイルを記録した場合はその旨を報告する。
- 各記事にタイトル、URL、要約があること。

## 報告

最終報告には、更新したデイリーノートのパス、対象ファイル数、`## 要約` セクションの新規作成または置換の別、注意点があればそれを簡潔に含める。
